# How to use Storage

Storage is the app where files live: uploads from humans, exports
from agents, fetched URLs, screenshots, generated artefacts. Before
calling any storage tool, hold these conventions.

## Mental model

- **Virtual folders, real paths.** `folder` is a string like
  `/contracts/2026/`. Always starts and ends with `/`. There's no
  real directory tree on disk — the path is metadata.
- **Files are addressed by id, never by path.** Two files in
  different folders can share a name; two files in the same folder
  can share a name. Always look up by id when you have one.
- **Bytes are deduped by sha256.** Uploading the same content twice
  returns the existing file's id with `was_existing: true`. Use
  `files_dedupe_check(sha256=…)` on purpose before a big upload.
- **Visibility is a security policy, not a UI flag.** Pick it
  carefully (see below).

## Folder conventions

| Root        | What goes there                                     |
|-------------|-----------------------------------------------------|
| `/inbox/`   | Default for human uploads with no folder hint.      |
| `/exports/` | Generated by agents (reports, CSVs, summaries).     |
| `/imports/` | Fetched from external URLs via `files_from_url`.    |
| `/archive/` | Older content the user explicitly moved aside.      |
| `/drafts/`  | Work-in-progress; safe to overwrite/delete liberally. |
| `/.media/`  | App-internal — thumbnails, transcodes. **Don't** show, share, or list these to users unless they ask explicitly. |
| `/.system/` | Reserved for future runtime use. Treat as off-limits. |

If the user gives you a folder that doesn't match these, follow
their wording. But default to one of the above when nothing's
specified.

## Visibility tiers

Three tiers. Default for new uploads is the install's
`default_visibility` setting (operator-set; see the storage app's
Settings panel).

| Tier      | Who can read                          | Use for                              |
|-----------|---------------------------------------|--------------------------------------|
| `private` | Authenticated apteva users only       | Anything internal. Default.          |
| `signed`  | Anyone with a time-limited HMAC URL   | External recipients, expiring shares.|
| `public`  | Anyone with the URL, forever          | Embeddable assets, public docs.      |

**Loosening privacy (private→signed, signed→public) requires
explicit user confirmation.** Quote the filename back. Tightening
is always safe.

## Tags vs folders

- **Folders** answer "where does this live?" — one folder per file.
- **Tags** answer "what is this about?" — many per file, freeform.

Use tags for cross-cutting attributes (project codes, priority,
status, audience). Use folders for the object's home. Don't tag
what you can fold; don't fold what you can tag.

## Search vs list

- `files_list(folder=…)` — directory-style. Cheap. Use when you
  know where to look.
- `files_search(q=…, tag=…, content_type=…)` — query-style across
  the whole project. Use when you don't.
- `files_dedupe_check(sha256=…)` — single hash lookup. Use before
  large uploads to skip them.

Default to `files_list` for "what's in <folder>", search for "find
me a file about X".

## Source field

Set `source` on every upload — the dashboard surfaces it.

- `human` — uploaded through the dashboard or a chat attachment.
- `agent` — generated by the agent itself (export, screenshot, summary).
- `imported` — fetched from an external URL via `files_from_url`.

Default to `agent` for tool-driven uploads unless context says
otherwise.

## Naming hygiene

- Keep extensions (`.pdf`, `.png`, `.csv`). The dashboard renders
  by MIME, but humans navigate by extension.
- Avoid spaces in names you generate yourself. Prefer
  `sales-q3-2026.csv` over `Sales Q3 2026.csv`. Human uploads keep
  whatever name the user chose.
- Don't include timestamps in names unless identity needs them
  (e.g. `db-backup-2026-05-03.tar.gz`). Storage tracks `created_at`
  for you.

## URLs you'll hand out

Three flavours. Pick the right one.

- **Content URL** (`/api/apps/storage/files/<id>/content`) — raw
  bytes, dashboard-cookie auth. Good for chat attachments and
  in-dashboard previews. Doesn't work for external recipients.
- **Signed URL** (from `files_get_url`) — same bytes, HMAC-signed
  query, expires after the TTL. Hand to external recipients.
- **Dashboard URL** (`/storage/files/<id>`) — metadata page. Use
  when the recipient is an apteva user and you want them to see
  the file in context.

## Soft vs hard delete

`files_delete` is **hard delete by default** — row removed AND
bytes removed. Pass `keep_record=true` for soft delete (row stays
for audit, bytes go on the next sweeper pass).

Confirm before hard-deleting unless the user said "delete forever",
"hard delete", "purge", "remove permanently". Default to asking.

## Don'ts

- Don't `files_set_visibility(public)` without confirming.
- Don't list `/.media/` or `/.system/` to the user.
- Don't share thumbnails — find the source file and share that.
- Don't loop `files_upload` for many URLs — `files_from_url` is
  the right primitive (dedupes, follows redirects, sets
  `source: imported`).

## When in doubt

Run `files_get` on the file you're about to act on. The
`visibility`, `folder`, `tags`, `source`, and `created_at` fields
are usually enough to make the right decision without asking.
